The 10 RAG Patterns
Complete Showcase

Five sub-projects. Ten retrieval patterns. One unified corpus. From simple vector search to autonomous agents and knowledge graphs — every pattern implemented, visualized, and interactive.

5 Sub-Projects 10 Patterns ~5,500 LOC ~215 Tests

The Big Insight: RAG is not just "vector search + LLM." It's about getting the right context, at the right time, in the right way. These 10 patterns represent the full spectrum of how production systems solve that problem.

01The 10 Patterns

P1
Foundations
Existing + Refactor
The retrieval baseline. Demonstrates the difference between naive vector search and the full hybrid pipeline, plus HyDE's query transformation technique.
#1 Simple RAG
Basic vector-only retrieval. Good for demos, weak for production. The starting point every system builds upon.
Query → Embed → Vector Search → Generate
#4 HyDE
Generate a hypothetical answer first, then use it to search. Bridges the structural gap between queries and documents.
Query → Hypothetical Doc → Embed Hypothesis → Search → Generate
P2
Conversational
New Build
Query augmentation patterns. Memory prepends conversation history; Branched decomposes complex questions into retrievable sub-queries.
#2 RAG + Memory
Adds conversation history to each retrieval. Essential for chatbots where context carries across turns.
History + Query → Augmented Query → Retrieve → Generate
#3 Branched RAG
Breaks complex questions into sub-questions, retrieves for each, then merges. Think: research assistant.
Query → Decompose → N Sub-Queries → N Retrievals → Merge → Generate
P3
Quality Control
New Build
Decision-making patterns. Adaptive decides whether to retrieve at all; Corrective evaluates quality and retries if results are poor.
#5 Adaptive RAG
Classifies query complexity: skip retrieval, simple search, or full pipeline. Saves cost and improves speed.
Query → Classify → [None | Simple | Complex] → Route → Generate
#6 Corrective RAG
Checks if retrieved data is relevant. If quality is low, expands the query and retries. A quality control layer.
Query → Retrieve → Quality Check → [Pass | Expand + Retry] → Generate
P4
Autonomous
New Build
Self-improving patterns. The LLM reasons about its own process — critiquing relevance, calling tools, and iterating until the answer is good.
#7 Self RAG
Model critiques itself: "Is this relevant? Do I have enough evidence?" More reliable for high-stakes use.
Query → Retrieve → Filter → Generate → Critique → [Retry | Accept]
#8 Agentic RAG
LLM acts as an agent: searches multiple times, calls tools, iterates. Where modern AI systems are heading.
Query → Think → Act (Search/Tool) → Observe → Repeat → Final Answer
P5
Advanced Structures
New Build
Beyond text. Multimodal handles images and tables. Graph RAG builds knowledge relationships for complex reasoning.
#9 Multimodal RAG
Handles images, tables, and charts by converting everything into searchable embeddings.
Image/Table → Caption/Parse → Embed → Unified Search → Generate
#10 Graph RAG
Builds a knowledge graph of entity relationships. Instead of "find similar text," it understands how things connect.
Docs → Extract Entities → Build Graph → Query → Traverse → Generate

02Architecture

Monorepo with Shared Library

All 5 projects live in one repo, sharing a rag_core Python package for embeddings, retrieval, reranking, and generation. Each project adds only its pattern-specific logic.

Docker Architecture

  • 1 shared Ollama container — all 5 services connect via internal Docker network
  • 1 shared model_cache volume — embedding + reranker models downloaded once
  • Separate ChromaDB collections per project within a shared volume
  • Corpus mounted read-only into every container
  • All ports bound to 127.0.0.1 only

Pros

  • Portfolio gold — No one showcases all 10 RAG patterns interactively. Standout differentiator.
  • Shared library extraction is overdue — Refactoring rag_core improves P1 regardless of the expansion.
  • Incremental delivery — Each phase ships a working project. Stop at any phase and still have value.
  • Fair comparison — Same corpus across all 5 makes pattern comparison meaningful.
  • No external API costs — Everything runs locally on Ollama. Zero inference bills.
  • Educational value — Each UI explains the pattern with pipeline visualizations. Teaching tool, not just code.

Cons & Risks

  • RAM pressure — All 5 running simultaneously = ~12.4 GB. Tight with other portfolio services.
  • Ollama bottleneck — Single instance, 4 vCPU. Multi-call patterns (Agentic, Self RAG) could hit 60-120s latency.
  • P5 is hardest — Vision models need ~4 GB extra. Graph NER with small models produces unreliable results.
  • Large scope — ~103 files, ~5,500 LOC. Risk of leaving it half-done.
  • Model quality — phi3:mini may not be "smart enough" for self-critique or query classification, making some demos look broken.
  • Infra complexity — 5 services need routing, monitoring, and CI/CD matrix builds.

Mitigations

RiskMitigation
RAM pressureUse Docker Compose profiles — run 1-2 services at a time, not all 5
Ollama bottleneckAccept demo latency. Add loading indicators. Consider mistral:7b for better reasoning.
P5 difficultySimplify: regex NER for Graph RAG, Ollama llava for images. Or defer P5 entirely.
Large scopeBuild P1-P3 first (6 patterns). P4-P5 can be Phase 2.
Model qualityTest LLM prompts early. Switch to mistral:7b if phi3:mini can't handle classification/critique.

Recommendation: Build P1–P3 first (6 patterns: Simple, HyDE, Memory, Branched, Adaptive, Corrective). These cover 80% of real-world RAG use cases and work reliably with phi3:mini. P4–P5 can be added when ready to upgrade the model or add RAM.

04Implementation Phases

Phase 1
Extract Shared Library
Create shared/rag_core package. Move services, schemas, logging, config, routers. Parameterize collection names. Make GeneratorService extensible.
Phase 2
Refactor P1 into p1-foundations/
Restructure existing code. Add SimpleRetriever (vector-only), HyDE service, mode toggle UI. Verify all 41 tests pass.
Phase 3
Build P2 — Conversational
Memory store, memory-augmented retrieval, query decomposition, branched retrieval with parallel sub-queries. Chat UI. ~35 tests.
Phase 4
Build P3 — Quality Control
Query classifier, adaptive router, quality checker, corrective retry loop. Decision flowchart UI. ~40 tests.
Phase 5
Build P4 — Autonomous
Self-critique pipeline, ReAct agent loop, tool registry. Agent trace visualization UI. ~45 tests.
Phase 6
Build P5 — Advanced Structures
Image processing, table extraction, multimodal retrieval. Entity extraction, NetworkX graph, graph traversal. d3.js graph UI. ~50 tests.
Phase 7
Integration & Polish
Root docker-compose.yml, Makefile, CI/CD matrix, README with project index, Nginx routing.
